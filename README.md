# Waterloo AI Project: Traffic Sign Detection

이 프로젝트는 YOLOv8을 활용하여 도로 위의 교통 표지판을 실시간으로 탐지하고 분류하는 시스템을 구축하는 것을 목표로 합니다. 학습된 모델은 서버-클라이언트 구조를 통해 통신하며, 라즈베리 파이와 같은 엣지 디바이스에서의 활용을 고려하여 설계되었습니다.

## 주요 기능
- **교통 표지판 탐지**: YOLOv8 모델을 사용하여 실시간 영상에서 표지판 위치 식별.
- **서버-클라이언트 아키텍처**: `detection_server.py`와 `detect_client.py`를 통한 분산 처리 구조.
- **데이터 전처리**: 원본 라벨 데이터를 YOLO 형식으로 자동 변환 (`label_to_yolo.py`).
- **모델 변환**: 추론 속도 최적화를 위한 PyTorch(.pt) 모델의 ONNX 변환 지원.

## 기술 스택
- **Model**: Ultralytics YOLOv8
- **Language**: Python 3.x
- **Libraries**: OpenCV, PyTorch, ONNX
- **Deployment**: Raspberry Pi (예정)

## 주요 파일 설명
- `yolov8_train.py`: YOLOv8 모델 학습 스크립트.
- `detection_server.py`: 소켓 통신을 이용한 객체 탐지 서버.
- `detect.py`: 메인 객체 탐지 추론 로직.
- `label_to_yolo.py`: 데이터셋 라벨 형식 변환 도구.
- `pt_to_onnx.py`: 모델 경량화 및 호환성을 위한 포맷 변환.

## 프로젝트 한계 및 개선 필요 사항

### 1. 모델 경량화 실패
- **문제점**: 라즈베리 파이 등 저사양 엣지 디바이스에서 실시간 추론을 수행하기 위해 ONNX 변환 및 경량화를 시도했으나, 기대했던 수준의 성능 향상을 얻지 못했습니다.
- **원인**: 모델 파라미터 최적화 부족 및 특정 하드웨어 가속(NPU/GPU) 활용의 제약으로 인해 FPS(Frames Per Second)가 실시간 서비스에 부적합한 수준으로 나타남.

### 2. 코드 로직상의 오류
- **문제점**: 특정 상황에서 서버와 클라이언트 간의 데이터 동기화 문제나 추론 결과 매핑 오류가 발생할 가능성이 있습니다.
- **상세**:
    - 네트워크 지연 발생 시 클라이언트의 영상 프레임과 서버의 탐지 결과 간의 타임스탬프 불일치 문제.
    - 예외 처리 로직이 미흡하여 통신 끊김 시 서버가 비정상 종료되는 현상.

### 3. 로봇 주행 및 제어 로직의 미흡함
- **문제점**: 센서 피드백이 결여된 단순 제어 방식으로 인해 주행 안정성이 떨어집니다.
- **상세**:
    - **Open-loop 제어**: 엔코더나 자이로 센서 피드백 없이 `time.sleep()`과 고정된 모터 속도에 의존하여 회전 및 직진을 수행하므로 외부 환경 변화에 취약함.
    - **단순한 라인 트레이싱**: PID 제어 없이 7가지 조건문(`if-elif`) 기반으로 조향을 결정하여 주행이 거칠고 라인 이탈 가능성이 높음.
    - **상태 관리 부재**: 표지판 인식에 따른 특수 동작과 일반 라인 트레이싱 동작 간의 우선순위 및 상태 전이가 명확하지 않아 동작 간 간섭 발생 가능.
    - **블로킹(Blocking) 코드**: 긴 시간의 `sleep` 사용으로 인해 동작 중 실시간 센서 데이터 처리가 불가능한 구간 존재.

## 향후 과제
- TensorRT 등을 활용한 추가적인 최적화 진행.
- 비동기 통신 방식을 도입하여 네트워크 지연에 따른 프레임 드랍 최소화.
- 데이터 증강(Augmentation)을 통한 탐지 정확도 향상.
